## Project Overview

A tamper-resistant digital evidence workflow that combines:

- **Ethereum (Ganache)** ‚Äî immutable on-chain evidence (same as before)
- **Solidity** ‚Äî smart contract (`contracts/ChainOfCustody.sol`)
- **Python + Flask** ‚Äî web backend (blueprints for API, help, features, admin)
- **SQLAlchemy + SQLite** ‚Äî relational data (assignments, audits, alerts, help topics, etc.)
- **Web3.py** ‚Äî contract calls and transactions
- **SHA-256** ‚Äî integrity hashing
- **cryptography (Fernet)** ‚Äî optional at-rest encryption for uploads
- **Bootstrap 5** ‚Äî responsive UI

---

## Roles and permissions

| Role | Capabilities |
|------|----------------|
| **Admin** | Full control: manage users and cases, assign evidence to investigators/members, approve access requests, view/edit/delete any data, set time locks |
| **Investigator** | Access **only cases assigned by Admin**; upload evidence, analyze case data, generate visualizations; cannot delete/modify system config or unauthorized cases |
| **Member** | **Basic user**: request access to a case (pending admin approval), view **only their own assigned cases**; no access to other users‚Äô data |

Investigators and Members get cases only through **case assignments** (Admin assigns evidence IDs to them). Members can also submit **case access requests** that Admins approve.

---

## Where data is stored

### 1. **Ganache / blockchain (unchanged model)**

- **Evidence records** (`addEvidence` on contract): hash, filename, type, size, uploader ‚Äî **on-chain** via the smart contract.
- **Custody events** (`logCustodyEvent`): action, actor, notes, timestamps ‚Äî **on-chain**.
- **Verification** (`verifyEvidence`): hash comparison ‚Äî **on-chain** (read-only).

So the **on-chain part is still the same**: one contract, one chain, same Solidity logic. Nothing here moved to JSON files.

### 2. **SQLite database**

Everything relational is in **SQLite** (SQLAlchemy), not JSON files:

| Data | Table |
|------|--------|
| Help topics | `help_topics` |
| Forensic cases & links | `forensic_cases`, `case_evidence_links` |
| Off-chain meta (local path, optional IPFS CID) | `offchain_evidence_meta` |
| Transfer requests | `transfer_requests` |
| Audit log (geo, HMAC signature) | `audit_log_entries` |
| Security alerts | `security_alerts` |
| Verification history | `verification_records` |
| Time locks | `evidence_time_locks` |
| **Case assignments** (Admin ‚Üí Investigator/Member) | `case_assignments` |
| **Case access requests** (Member ‚Üí pending/approved) | `case_access_requests` |

**Summary:** **Blockchain evidence stays on Ganache**; **everything else is in SQLite** (not JSON). The app uses **both**: chain + database ‚Äî chain for immutability, DB for roles, assignments, audits, graphs, etc.

## Setup Instructions

### 1. Install Ganache

Download **Ganache** from [Truffle Suite](https://trufflesuite.com/ganache/), start a workspace (e.g. Quickstart), and confirm it listens on **port 7545** (default for Ganache GUI).

### 2. Python environment

From the project root (`fyp/`):

```bash
python3 -m venv .venv
source .venv/bin/activate          # Linux / macOS
# .venv\Scripts\activate           # Windows

python3 -m pip install --upgrade pip
pip install -r requirements.txt
```

### 3. Deploy the smart contract

From the project root:

```bash
python deploy.py
```

This installs the Solidity compiler (**solc 0.8.7**, as configured in `deploy.py`), compiles the contract, deploys it to Ganache, and writes:

- `backend/contract_abi.json`
- `backend/contract_address.txt`

### 4. Run the Flask app

```bash
cd backend
python app.py
```

Open **http://127.0.0.1:5000** in your browser.

## How it works

### Evidence upload

1. User uploads a file (allowed types are enforced in `app.py`).
2. The server computes **SHA-256** of the original bytes.
3. Optionally the file is **encrypted** with Fernet and saved under `uploads/`.
4. Hash and metadata are sent to the contract via **`addEvidence`**.
5. The transaction is mined on Ganache; the record is append-only on-chain.

### Verification

1. User supplies an evidence ID and a file.
2. A new hash is computed and checked with **`verifyEvidence`**.
3. A **Verified** custody event can be logged on-chain.

### Custody chain

Actions (e.g. transfer, analyze, verify) are recorded with **`logCustodyEvent`**, producing an immutable timeline per evidence ID.

### Help pages

After login, **Help** in the navbar links to topic pages. Copy is stored in **SQLite** (`help_topics` table) and loaded through **SQLAlchemy**; the database is initialized when the app starts.

### Role-based access

- **Admin** sees all evidence and cases; can assign evidence to investigators/members; approves case access requests.
- **Investigator** sees only evidence IDs listed in **`case_assignments`** for their username.
- **Member** sees only evidence IDs assigned to them (same table); can submit **case access requests** for approval.

### Admin panel

**Admin** users can open **Administration** in the navbar to:

- Manage users and roles (demo: edit `users_config.py` or use the admin users form)
- Assign blockchain evidence IDs to investigators or members
- Approve or reject **case access requests** from Members

|--------|----------|------|-------------|

| `GET` | `/api/evidence` | Yes (session) | List evidence summaries |

Use a logged-in browser session or mirror the same cookie when calling protected routes.

## Smart contract (summary)

| Function | Role |
|----------|------|
| `addEvidence(...)` | Register new evidence metadata and hash |
| `logCustodyEvent(...)` | Append a custody event |
| `verifyEvidence(id, newHash)` | Compare stored vs new hash |
| `getCustodyChain(id)` | Full event list for an item |
| `getEvidence(id)` | Metadata for one item |
| `getEvidenceCount()` | Total count |
